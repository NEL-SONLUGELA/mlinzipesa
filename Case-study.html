MlinziPesa
Case study

A fraud shield and forensic ledger built for Tanzania's mobile money economy

MlinziPesa protects small and medium businesses from SMS-based mobile money fraud through real-time message screening, live payment verification, and a tamper-evident audit trail — designed for the realities of Swahili and English smishing, sub-2-second verification needs, and TRA compliance.

The problem

Mobile money fraud is outpacing the businesses that depend on it

Tanzania's mobile accounts grew from 35.5 million in 2021 to 76.5 million in 2025, with transactions projected to reach 6.3 billion. That growth has been matched by a surge in smishing, fake payment receipts, and SIM-swap fraud — and SMEs, who rely on manual verification of SMS receipts, are the most exposed.

The approach

Three layers of defense, built around how a transaction actually happens

Rather than one generic fraud filter, MlinziPesa is structured around the three points where a fraudulent transaction can be caught: before the message is trusted, before the payment is recorded, and after — when it needs to be proven.

01
SMS smishing scanner
On-device NLP scans incoming SMS in Swahili and English for known fraud patterns — like "TumaPIN" requests or fake TRA codes — before the cashier ever reads them.
02
Real-time verification
Every claimed payment is checked against the operator's API sandbox in under two seconds, catching fake receipts before they're treated as paid.
03
Forensic ledger
Verified payments are written into a hash-chained, tamper-evident log — exportable as PDF or Excel for TRA audits.

System architecture

How a payment moves through the system

  1. 01Customer pays via mobile money; an SMS receipt arrives at the cashier's device.
  2. 02On-device NLP scans the SMS text for known smishing patterns before the cashier sees it.
  3. 03If safe, the verification module checks the claimed amount against the operator API sandbox in real time, target under two seconds.
  4. 04A match writes a new block to the forensic ledger, hashed against the previous block's hash.
  5. 05A mismatch is flagged immediately and excluded from the ledger until a human reviews it.
  6. 06Owners view live totals on the dashboard; auditors export the ledger as PDF or Excel for TRA compliance.

Methodology

How it was built

Requirements & analysis
Engaged a pilot group of 50 SMEs to understand vulnerabilities, transaction volume, and accounting needs; reviewed operator API docs and TRA compliance requirements.
System design
Designed the NLP engine, the verification module's API integration, and the forensic ledger's hashing mechanism.
Implementation
Built the cashier-facing interface, the NLP scanning logic, and the backend ledger, focused on performance and security.
Testing & evaluation
Unit, integration, and pilot testing against the 60% smishing-reduction and sub-2-second latency targets.

This case study presents the design and prototype of MlinziPesa. The interactive prototype demonstrates the intended user flow using sample and rule-based detection logic; production deployment would require a trained NLP model, live integration with mobile operator APIs, and cryptographic (not illustrative) hashing for the ledger.
